# RFC：模块资源层（Module Resource Layer）

状态：草案

## 摘要

Web Widget 已通过 `@route`、`@widget`、`@action`、`@layout`、`@middleware` 等文件约定，为源码模块赋予框架语义。这些约定降低了常见场景的使用成本，但每种语义目前仍由框架分别识别、构建、登记并消费。新增一种文件角色时，通常需要同时进入文件扫描、构建入口、模块转换、manifest、开发服务器和运行时等多个环节，扩展边界并不清晰。

本 RFC 提议引入**模块资源层（Module Resource Layer，MRL）**这一概念：在源码模块与 Web Widget 的构建、开发服务器和运行时设施之间，建立统一的语义抽象层。源码文件、模块导出、配置或模块使用方式可以被识别为一种或多种**模块资源**；模块资源通过自身的**种类**与**能力**描述其框架语义，再由对应的消费者处理。

模块资源层灵感来自于 Linux 虚拟文件系统的核心思想：统一的是资源模型、能力表达和分派边界，而不是强迫所有资源拥有相同的行为。它不替代 ESM、Vite 插件系统或现有运行时协议，也不要求立即改变当前文件命名约定。

从这个角度看，模块资源层是**抽象的抽象**：它不直接规定客户端边界、可恢复性、数据同步等问题的唯一答案，而是为这些框架抽象提供共同的发现、描述、接入和消费方式。未来的 Web 框架不仅需要降低业务开发的复杂度，也需要控制自身不断增加新抽象所带来的复杂度。

本 RFC 只讨论概念模型、职责边界和设计原则，不规定公开 API、配置格式、manifest 结构或具体迁移步骤。

## 动机

### 优秀抽象为何难以跨越技术栈

Web 世界从来不缺少好的抽象。React 提出的 `'use client'` 和 `'use server'` 就是很有代表性的例子：一行看起来近乎普通的字符串，清楚地表达了客户端与服务端的边界。开发者不必直接面对模块图如何切分、引用如何跨越边界、代码最终进入哪个构建产物，这些复杂度由 React RSC 及其配套工具链承担。

但这份抽象的力量并不真的来自那行字符串。它来自字符串背后的 React 组件模型、编译转换、模块引用协议和运行时。把 `'use client'` 放进一个 `.vue` 文件，并不会让它自然变成 React 意义上的 Client Component。这里没有谁的设计做错了；恰恰相反，`'use client'` 之所以好用，是因为 React 为它完成了一个非常完整的语义闭环。也正因为闭环如此完整，它很难脱离原来的技术栈独立存在。

Qwik 的可恢复性也是如此。它让页面不必先执行传统意义上的整棵组件树水合，而可以在更细的粒度上恢复交互，`$` 标记则把这种能力带到开发者看得见的函数边界上。但其他框架并不能仅仅采用相同的函数命名约定就获得可恢复性；真正工作的是 optimizer、序列化模型、事件恢复和运行时之间的整体协作。`$` 是这个抽象最醒目的入口，却不是抽象的全部。

所以我们并不是认为这些框架抽象本身有什么问题。它们都在各自的约束下给出了有价值的答案。问题是，今天的 Web 框架善于发明抽象，却缺少一种承载抽象的共同方式。一个新概念经常同时绑定自己的源码语法、编译规则、构建插件、manifest 和运行时，于是很难被另一种技术栈实现，也很难被框架之外的工具理解。

### 局部最优如何累积为框架复杂度

即使拥有 `'use client'` 和 `'use server'`，建立在 React 上层的元框架仍然需要文件路由、特殊文件名、数据加载、缓存和部署约定。Web 开发框架最终仍是许多局部最优方案的组合：每一项单独看都合理，组合起来却意味着框架必须维护越来越多彼此独立的识别、编排和兼容逻辑。这也是现代框架复杂度的重要来源之一。

传统意义上的框架主要帮助开发者降低业务开发的复杂度。但当框架持续吸收路由、SSR、数据同步、代码分发和部署能力时，它也需要一种机制来降低自身的复杂度。可扩展性不能只意味着以后还能再增加一个 API；如果每个 API 都要穿透文件扫描、编译、manifest、开发服务器和运行时，那么功能越多，框架反而越难继续演进。

### Web Widget 缺少的共同语义落点

Web Widget 当前的文件约定正是这个问题的一个缩影。`@route`、`@widget`、`@action`、`@layout` 和 `@middleware` 为模块角色提供了非常有效的上层语言：角色一眼可见，文件容易搜索，常见项目无需逐项配置，构建工具也能在加载模块之前完成初步识别。问题不在于这些约定需要被移除，而在于通过文件名发现角色与角色在整个框架中的语义目前仍然紧密绑定。一个新的文件角色不仅要被扫描器认识，还往往需要分别进入构建入口、模块转换、routemap、开发服务器和运行时；增加一种约定，实际上是在多个子系统中增加一条特殊路径。

项目已经通过 `WebWidgetAdapter` 解决了另一个维度的扩展问题：React、Vue、Svelte 或其他框架组件可以被转换到共同的 Widget 渲染协议。但源码所属的 UI 框架与模块在 Web Widget 中承担的角色是两个正交的问题。同一个 route 可以由不同 UI adapter 提供渲染能力，同一个 React 模块也可能根据使用方式成为普通组件、route 或 widget。当前架构已经拥有 Vite 插件、模块 schema、routemap、Widget adapter 和内部插件 host，却还没有一个共同的领域对象，把 action 散落在这些设施中的不同表现归一为同一资源。我们需要的不是再增加一个构建系统，而是为现有设施补上这个语义落点。

### 数据同步背后的资源语义

前后端同构应用中的数据获取把这个矛盾表现得很具体。一个组件直接调用原生 `fetch` 看起来非常自然，直到同一组件先在服务端渲染、随后又在浏览器激活。SSR 和 Hydration 是两个独立生命周期，它们甚至可以看作运行在两只不同的时钟上：服务端第一次请求数据时得到的是 A，浏览器稍后请求时可能已经变成 B。时间戳会变化，用户权限上下文可能不同，数据也可能恰好在两次请求之间更新。浏览器此时面对的是由 A 生成的 HTML，却试图用 B 完成首次渲染，hydration mismatch 和重复请求便很容易出现。

现代元框架通常会在原生 `fetch` 之上补充自己的数据生命周期。有的增强 `fetch`，有的提供 loader、服务端数据函数或异步数据 composable，有的把状态显式序列化到浏览器。它们的 API 各不相同，但都在处理同一件事：让一次服务端请求中产生的数据能够被识别和复用，把允许暴露的结果传递到浏览器，让首次客户端渲染读取与 SSR 相同的快照，然后再进入正常的失效、刷新或重新请求阶段。

这些 API 都很有用，但 Web Widget 的定位是框架的框架。如果我们为 React 提供一个 hook、为 Vue 提供一个 composable、再为其他技术栈提供各自版本，解决方案仍然被锁在局部技术栈中，我们也只是在框架内部复制同一种问题。

换一个角度，这个问题描述的其实是一种资源：它在服务端收到请求之后产生，可以在当前请求中缓存和复用，其中允许暴露的部分可以被带到浏览器，并至少持续到浏览器完成首次渲染。组件使用什么语法消费它是次要的，真正稳定的是这段生命周期和两端共享同一份结果的约束。

当我们把这项语义理解为资源以后，`@resource` 文件就成为一种自然且容易理解的表达。文件负责描述资源是什么以及如何产生；生命周期缓存、序列化、传输和客户端复用由框架完成。React、Vue 或其他组件消费的是同一种资源，不需要先共同接受某个框架专用的数据获取 API。

`@resource` 在这里仍然只是一个帮助理解的例子，而不是本 RFC 已经决定的新文件约定。项目现有的 `lifecycleCache`、`cacheProvider` 和 `syncCacheProvider` 已经在做其中重要的一部分：缓存从服务端收到请求开始，可以被序列化并延续到客户端。这证明跨组件技术栈的生命周期语义是可以成立的。由此自然产生了更一般的追问：这样的语义能否不再只是框架预先设计好的一组函数，而成为一种可被发现、描述、组合和扩展的模块资源？

### 从资源示例回到框架本身

同样的追问不只适用于数据。未来还可能出现定时任务、消息消费者、静态内容、邮件模板，或者今天尚未想到的模块角色。它们可以由文件名发现，也可能由模块的使用方式、显式配置、静态元数据或上层工具生成。发现方式可以不同，下游看到的却应是同一种稳定的资源语义，而不必再次根据文件名猜测它是什么。

把这些不同来源归一为资源仍然只是开始。资源还必须说明自己的语义由谁负责，在哪些阶段发生作用，又由哪些框架设施实现。同一个资源在开发、生产构建和运行时中也应保持一致身份，而不是由每个阶段重新解释。否则所谓扩展仍然只是把特殊分支从一个地方搬到了另一个地方。

我们缺少的正是这样的**抽象的抽象**。它不试图把 React、Vue、Qwik 的所有能力改造成同一个最小 API，也不规定客户端边界、可恢复性或数据同步的唯一答案。它要回答的是一件更基础的事：**一个框架如何容纳一种新的抽象，让它拥有可发现的身份、明确的规则和完整的实现边界，而不要求整个框架为它增加一条新的特殊路径。**

如果传统框架的使命是简化业务开发，那么未来的 Web 框架还需要在不断创造新抽象的同时，也能够简化自身。

## 目标

- 为 Web Widget 中具有框架语义的源码模块建立统一概念模型。
- 让模块角色从核心代码中的封闭枚举转变为可扩展集合。
- 分离资源发现方式、资源语义和资源消费方式。
- 允许不同资源通过能力表达参与不同框架流程，而不要求所有资源实现相同行为。
- 保持 UI 框架适配、模块角色、构建工具和运行时消费者之间的清晰边界。
- 为文件约定、调用方推导和显式配置等识别方式提供共同下游。
- 为开发、构建、manifest、资产和运行时建立一致的资源身份与诊断语境。
- 使内置资源与第三方资源遵循相同的核心模型。

## 非目标

- 不在本 RFC 中定义 TypeScript 接口或公开配置格式。
- 不在本 RFC 中决定模块资源如何注册、加载或排序。
- 不设计新的 JavaScript 模块加载器，也不替代 ESM。
- 不复制 Vite 的 resolve、load、transform、模块图或 HMR 能力。
- 不把 routemap、importmap 或其他现有 manifest 立即合并为万能 manifest。
- 不要求所有模块资源支持路径、渲染、客户端代码或运行时执行。
- 不以移除 `@route`、`@widget`、`@action` 等文件约定为目标。
- 不承诺任意第三方资源都能被 Web Router 或 Widget runtime 自动理解。
- 不在概念尚未稳定时确定兼容周期和迁移阶段。

## 概念模型

### 模块资源层

模块资源层是位于源码世界与框架设施之间的语义层：

```text
源码文件 / 模块导出 / 使用方式 / 配置
                    │
                    │ 发现与识别
                    ▼
              模块资源层（MRL）
                    │
                    │ 按种类和能力分派
                    ▼
       开发服务器 / 构建 / 资产 / manifest / 运行时
```

它将上游多样的识别信号归一为模块资源，再将资源交给理解其能力的下游消费者。模块资源层本身不实现所有能力，而是维护稳定的表达和分派边界。

### 术语

| 术语                                   | 含义                                                                               |
| -------------------------------------- | ---------------------------------------------------------------------------------- |
| 源码模块（Source Module）              | ESM、框架组件或其他可被构建工具处理的原始模块。它是资源的载体，但不等同于资源。    |
| 模块资源（Module Resource）            | 源码模块在 Web Widget 领域中的一个语义身份，例如 route、widget 或 action。         |
| 资源种类（Module Kind）                | 标识资源所属的语义类别，并确定哪些约束和能力对它有意义。                           |
| 资源能力（Module Capability）          | 描述资源能够参与的框架行为，例如可寻址、可渲染、服务端执行、客户端引用或拥有资产。 |
| 资源提供者（Module Resource Provider） | 负责从某类输入中发现资源，并对该资源种类的语义描述负责。                           |
| 资源消费者（Module Resource Consumer） | 理解某项能力或某类资源，并将其接入构建、开发服务器或运行时行为的一方。             |
| 资源注册表（Module Resource Registry） | 表达当前系统认识哪些提供者、消费者、种类与能力的逻辑边界。                         |
| 资源图（Module Resource Graph）        | 描述资源身份及其相互引用关系的概念视图；是否形成具体持久化数据结构由后续设计决定。 |

### 与虚拟文件系统的类比

模块资源层的设计灵感来自 VFS，但不照搬文件系统接口：

| Linux VFS              | 模块资源层                                 |
| ---------------------- | ------------------------------------------ |
| VFS 统一抽象层         | Module Resource Layer                      |
| 文件系统中的对象身份   | Module Resource                            |
| 文件系统类型或驱动     | Module Resource Provider                   |
| 对对象可执行的操作集合 | Module Capability                          |
| VFS 对操作的分派       | Consumer 对资源能力的分派                  |
| 挂载后形成的命名空间   | Registry / Resource Graph 所表达的资源空间 |

类比的重点是：VFS 没有让普通文件、目录、设备和 socket 具有完全相同的行为，而是提供统一身份和操作边界，并允许具体实现声明自身支持的能力。模块资源层也不应寻找所有角色的最低公共行为，而应允许不同资源拥有不同能力组合。

类比的边界同样重要。Web Widget 不提供通用的文件读写、权限、目录或挂载语义，因此不应把这一层命名为文件系统，也不应为追求形式相似而引入与领域无关的操作。

## 设计原则

### 文件不等于资源

源码文件是物理组织形式，模块资源是框架语义。两者通常有关联，但不应被视为一一对应：

- 一个文件可以不形成任何模块资源；
- 一个文件可能形成一个或多个模块资源；
- 一个资源未来也可能由生成信息或多个源码模块共同描述。

因此，文件路径可以参与资源身份构造，但不应成为资源的全部定义。

### 约定是发现机制，不是协议本身

`@route`、`@widget` 和 `@action` 应被理解为内置的资源发现语言。它们提供良好的默认体验，但下游行为应依赖被发现的资源语义，而不是重复匹配文件名。

这一原则允许约定继续存在，也允许某些角色将来采用调用方推导或显式声明，而不会形成两套构建与运行时管线。

### 种类表达身份，能力表达行为

资源种类回答它是什么，资源能力回答它能参与什么。两者不能互相替代。

仅使用种类分派容易重新形成不断增长的 `switch`；仅使用能力又可能丢失领域约束，使不同语义被错误地视为可互换对象。模块资源层应同时保留二者：种类提供语义所有权，能力提供跨种类复用的行为边界。

### 能力应按需组合

route 可能可寻址、可渲染并拥有客户端资产；action 可能可寻址、服务端执行并向客户端暴露代理；layout 可能可渲染但不独立寻址。模块资源层不应要求它们实现统一而庞大的生命周期。

消费者只应依赖自己理解的能力。资源没有声明的能力，不得由消费者根据文件名或历史惯例隐式推断。

### 扩展必须同时声明生产者与消费者责任

识别出一种新资源，不等于系统已经支持这种资源。资源提供者必须能够说明它产生什么语义；相关扩展还必须提供或指定理解这些语义的消费者。

当系统发现无人消费的资源、无法满足的必需能力或相互冲突的解释时，应产生明确诊断，而不是静默忽略或回落到偶然行为。

### 内置能力与扩展能力遵循同一模型

route、widget、action 等内置角色不应长期拥有绕过模块资源层的特权路径。否则第三方扩展只能使用模型的子集，核心中的特殊分支仍会继续增长。

这不意味着首个版本必须公开所有内部能力，而是要求最终架构能够用同一套概念解释内置行为和扩展行为。

### 复用宿主工具，不建立平行构建系统

Vite 已经负责模块解析、加载、转换、依赖图和热更新。模块资源层应表达 Web Widget 特有的语义和贡献，再由现有工具执行。它不应发展出另一套通用插件钩子来镜像 Vite。

类似地，Web Router、Widget runtime 和 adapter 仍然拥有各自的领域行为。模块资源层负责把合适的资源交给它们，不负责吞并它们。

### 语义必须跨阶段稳定

开发服务器、生产构建和运行时可能拥有不同信息与执行方式，但它们对资源身份、种类和已声明能力的理解必须兼容。阶段差异应表现为不同消费者对同一资源的处理，而不是每个阶段独立识别出不同角色。

### 确定性优先于隐式便利

资源发现、冲突处理和消费者选择必须具有可解释、可复现的结果。扩展性不能以插件顺序偶然决定语义为代价。任何优先级、覆盖或组合规则都应在后续协议中被明确表达。

## 职责边界

### 模块资源层负责

- 接收来自文件约定、使用方式、配置或其他来源的资源描述；
- 维护资源身份、种类、能力和必要的领域元数据；
- 将资源分派给理解相应能力的消费者；
- 检测重复身份、语义冲突、缺失消费者和不满足的能力；
- 为不同阶段提供一致的资源视图和诊断语境；
- 允许核心之外定义新的资源种类与配套消费者。

### 模块资源层不负责

- 解析或执行 JavaScript 模块；
- 编译 JSX、Vue SFC 或其他框架语法；
- 定义所有资源共享的运行时生命周期；
- 自动赋予未知资源业务含义；
- 将所有 manifest 强制归并为单一格式；
- 决定 Web Router、Widget runtime 或第三方运行时的内部实现；
- 保证任意能力组合都有意义。

## 与现有概念的关系

### 文件约定

现有文件约定可以作为模块资源层的内置提供者继续工作。对用户而言，常见项目仍可通过文件命名获得零配置体验；变化只发生在框架内部如何表达和分派这些语义。

### WebWidgetAdapter

`WebWidgetAdapter` 描述源 UI 框架如何提供通用渲染能力。模块资源层描述模块角色及其能力。二者构成两个正交维度：

```text
              模块角色
          route / widget / ...
                   ▲
                   │
源码框架 ───────────┼─────────── Web Widget 行为
React / Vue / ...  │
                   ▼
              框架适配能力
```

模块资源层不应复制 adapter 的框架识别与渲染职责；adapter 也不应决定一个模块是否是 route、action 或未来的新角色。

### 模块 schema

模块 schema 定义某类模块在阶段边界上的数据或调用契约。模块资源层可以引用这些契约来描述能力，但不取代 schema。资源表达模块承担的角色，schema 表达承担该角色时必须满足的格式。

### 生命周期缓存

`lifecycleCache`、`cacheProvider` 和 `syncCacheProvider` 是现有的框架中立运行时能力：缓存从服务端收到请求开始，可以序列化并传递到客户端，并已能在 React、Vue 等不同组件模型中使用。

它与模块资源层处于不同层次。生命周期缓存负责值的保存、暴露和复用；模块资源层负责描述某个模块定义了一项具有这种生命周期的数据资源，并把它交给相应消费者。前者可以作为后者未来可能复用的底层能力，但 MRL 不依赖必须采用当前缓存 API，也不把 `@resource` 的具体行为提前固定为现有实现。

### Routemap 与 importmap

routemap 和 importmap 是特定消费者需要的交付形式，不等同于模块资源层本身。模块资源层可以成为生成或解释这些数据的共同语义来源，但不预设它们必须合并，也不假设所有资源都应进入其中。

## 概念示例

假设生态扩展希望引入 `task` 资源，用于表达由特定运行时调度的后台任务。

在现有模式下，增加 `@task` 很容易退化为：核心扫描器认识一个新后缀，核心构建器增加一种入口，核心 manifest 增加一个字段，核心运行时再增加一条分支。此时文件约定扩展了，但框架核心也被同步扩张。

在模块资源层中，期望的责任关系是：

1. task 提供者负责从约定或其他信号中发现 task 资源；
2. task 资源声明其身份、领域元数据和所需能力；
3. task 的构建消费者负责解释与构建相关的能力；
4. task 的运行时扩展负责消费其交付信息；
5. 核心模块资源层只负责统一登记、分派、冲突检测和诊断。

核心不需要理解 task 如何调度，正如 VFS 不需要理解每个设备的业务用途。反过来，如果项目中没有安装理解 task 的消费者，仅仅出现一个 `@task` 文件也不应被视为完整支持。

这个示例用于检验职责边界，不代表本 RFC 提议增加 `@task`，也不预设 task 的构建或运行时形式。

## 架构不变量

后续设计应维持以下不变量：

1. **单一语义来源**：同一阶段中的消费者不应通过各自的文件名规则重复判断资源角色。
2. **显式能力**：消费者不得依赖资源没有声明的能力。
3. **开放种类**：核心代码不应以封闭联合类型或固定分支作为所有资源种类的最终边界。
4. **责任闭合**：新资源必须存在对其语义负责的提供者和消费者，核心不兜底猜测。
5. **阶段一致**：dev、build 和 runtime 对资源身份及基本语义的理解必须兼容。
6. **确定性**：相同输入和配置必须产生相同的资源集合、冲突结果与消费者选择。
7. **可诊断性**：未知种类、冲突解释、能力缺失和无人消费必须能够定位到具体资源及其来源。
8. **内外一致**：内置角色最终应能用与第三方角色相同的概念模型解释。
9. **宿主复用**：通用模块处理能力继续由 ESM、Vite 和底层构建工具提供。
10. **渐进兼容**：引入抽象本身不应要求用户立即重命名文件或改变现有运行时行为。

## 成功标准

模块资源层是否有价值，不取决于是否出现了新的通用接口，而应由以下结果判断：

- 能够在核心包之外定义一个新的模块种类及其消费者；
- 接入新种类时，无需在核心中追加角色枚举、marker 列表或按种类分支；
- 新资源能在开发与生产构建中保持一致身份并获得明确诊断；
- 内置 route、widget、action 等角色可以由同一概念模型完整解释；
- 文件约定可以替换或补充，而不复制下游构建与运行时管线；
- 模块资源层没有演化为对 Vite 插件系统的重复封装；
- 不支持的能力组合能够在进入不确定运行时行为前失败。

一个有意严格的验收场景是：由核心仓库之外的扩展定义一种示例资源，在不修改核心角色列表和特殊分支的前提下完成发现、构建期交付、开发期更新和专属运行时消费。该场景应在协议进入稳定状态前成立。

## 权衡与风险

### 过早抽象

当前内置角色数量有限，它们的共同模型可能尚未完全显现。若过早冻结公开协议，容易把现有实现偶然形成的步骤误认为长期语义。

因此，本 RFC 先确立术语、边界和不变量，不立即确定接口。后续设计应通过多个性质明显不同的内置角色以及至少一个外部示例资源验证抽象。

### 最低公共抽象

如果只抽取所有角色都具备的字段，模块资源层可能退化为带 `kind` 的文件路径记录，无法真正承载扩展语义。能力模型必须足以表达跨阶段贡献，同时避免把每种角色的全部细节提升为核心概念。

### 万能核心的诱惑

另一种风险是把每个新需求都标准化为核心能力，最终形成庞大的框架内核。能力是否进入公共词汇，应取决于是否存在跨资源种类、跨消费者的稳定语义，而不是单个扩展是否需要它。

专属语义应由资源种类及其消费者拥有；只有真正通用的行为才应成为共享能力。

### 顺序与冲突

多个提供者可能对同一个源码模块给出不同解释，多个消费者也可能声明处理相同能力。若结果依赖插件安装顺序，会削弱可预测性。

后续协议必须明确组合、排他、覆盖和冲突规则，但本 RFC 不提前选择具体策略。

### 跨阶段信息不对称

开发、构建和运行时可获得的信息不同。一个看似统一的资源对象可能诱使各阶段依赖实际上不可用的信息。

模块资源层应统一稳定语义，而不是假设所有阶段共享同一内存对象或完整数据。哪些信息属于资源身份、哪些只是阶段产物，需要在后续设计中继续区分。

### 兼容与版本

一旦第三方可以定义资源种类和能力，核心、提供者与消费者之间就会形成兼容关系。协议版本、能力演进和诊断策略将成为必要问题，但现在确定版本格式仍为时过早。

## 被刻意推迟的设计

以下问题会显著影响实现和公开 API，但不足以改变本 RFC 的概念判断，因此留待后续 RFC：

- 模块资源协议最终位于哪个包；
- 提供者和消费者使用对象协议、Vite plugin API 扩展还是其他注册方式；
- 资源身份如何生成，以及一个源码模块能否产生多个同种资源；
- 种类与能力是否允许命名空间，如何避免生态命名冲突；
- 能力由核心枚举、开放注册还是二者结合；
- 多个提供者命中同一模块时如何组合或报错；
- 多个消费者理解同一能力时如何选择；
- 资源图是临时概念视图、内存中间表示还是可观察的公开产物；
- 现有 routemap、importmap 与构建 manifest 如何从资源语义派生；
- 开发服务器中的失效粒度和更新传播如何表达；
- 扩展如何提供运行时消费者，以及如何进入 server entry；
- 协议兼容性、能力协商和版本管理方式；
- 内置角色迁移到统一模型的顺序与兼容周期；
- 哪些能力足够稳定，可以成为跨种类的公共词汇。

这些问题应在概念模型被真实角色验证后逐项收敛，不应在第一份 RFC 中由某个接口草案隐式决定。

## 开放问题

1. 一个源码模块同时形成多个资源时，它们应共享身份还是保持完全独立？
2. route 与 fallback 这样的关系应表达为不同种类、同一种类的不同元数据，还是能力组合？
3. `layout` 的单例或层级约束属于资源种类、消费者还是上层应用模型？
4. widget 的身份由目标文件、`widget()` 调用点还是二者共同决定？
5. 哪些行为已经足够稳定，可以成为首批公共能力？
6. 第三方运行时消费者应如何参与 server entry，同时不让模块资源层承担应用启动编排？
7. 资源图应只用于内部统一语义，还是应成为调试工具和生态插件可观察的接口？
8. 如何证明 dev 与 build 使用的是同一资源语义，而不是两套相似但独立的发现结果？

## 备选命名

### 虚拟模块系统（Virtual Module System）

未采用。Vite 生态中的 virtual module 通常指没有直接磁盘文件、由插件提供的模块 ID。该名称容易让人误以为本 RFC 在设计模块加载或虚拟模块生成机制。

### 模块系统（Module System）

未采用。该名称通常指 ESM、CommonJS、模块解析和加载规则，范围过大且与现有标准概念冲突。

### 约定系统（Convention System）

未采用。文件约定只是资源发现方式之一，该名称会继续把资源语义绑定在命名规则上。

### 资源图（Resource Graph）

未采用为顶层名称。资源图适合描述模块资源之间关系的概念视图或内部表示，但不能覆盖提供者、消费者、能力分派与诊断等完整职责。

### 模块资源系统（Module Resource System）

可作为非正式称呼，但层更准确地表达它位于源码模块与现有框架设施之间，并且不试图替代这些设施。
